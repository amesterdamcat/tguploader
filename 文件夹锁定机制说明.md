# 文件夹锁定机制说明

## 功能概述

为解决多个Telegram账号同时上传时可能选择同一个子文件夹导致资源浪费的问题，我们实现了一个基于文件锁的同步机制。

## 核心功能

### 1. 锁定文件格式

在每个正在上传的文件夹中创建 `.uploading.lock` 文件，包含以下信息：

```json
{
  "pid": 12345,
  "session_name": "account1_session", 
  "phone": "+1234567890",
  "start_time": "2025-09-13T11:01:50.790925",
  "folder_path": "/path/to/folder"
}
```

### 2. 核心方法

#### `_create_folder_lock(folder_path)`
- 为指定文件夹创建锁定文件
- 记录当前进程ID、账号信息和开始时间
- 使用原子写入操作确保数据完整性

#### `_check_folder_lock(folder_path)`
- 检查文件夹是否被其他进程锁定
- 验证锁定进程是否仍在运行
- 自动清理僵死锁文件（进程不存在或超时6小时）

#### `_remove_folder_lock(folder_path)`
- 移除文件夹锁定文件
- 在文件夹处理完成或程序异常退出时调用

#### `_select_available_folders(folder_groups)`
- 从所有文件夹中选择未被锁定的可用文件夹
- 跳过已被其他账号锁定的文件夹
- 返回随机打乱的可用文件夹列表

### 3. 工作流程

1. **启动检测**: 程序启动时检查所有子文件夹的锁定状态
2. **智能选择**: 只选择未被锁定的文件夹进行上传
3. **锁定保护**: 开始处理文件夹时立即创建锁定文件
4. **自动清理**: 文件夹处理完成后自动移除锁定文件
5. **异常处理**: 程序异常退出时确保清理锁定文件

### 4. 安全特性

- **僵死进程检测**: 自动检测并清理不存在进程的锁文件
- **超时清理**: 超过6小时的锁文件视为僵死文件并清理
- **原子操作**: 使用JSON格式和原子写入确保数据完整性
- **异常安全**: 在try-finally块中确保锁文件被正确清理

## 使用效果

### 原有问题
```
账号1: 随机选择文件夹A -> 开始上传
账号2: 随机选择文件夹A -> 重复上传 ❌
```

### 解决后效果  
```
账号1: 检查锁定 -> 锁定文件夹A -> 开始上传
账号2: 检查锁定 -> 发现文件夹A被锁定 -> 选择文件夹B -> 开始上传 ✅
```

## 技术依赖

- `psutil`: 用于进程存在性检测
- `json`: 锁定信息序列化
- `datetime`: 时间戳记录
- `os`: 文件操作

## 性能影响

- **最小开销**: 仅在文件夹级别添加简单的文件检查
- **无阻塞**: 不使用阻塞锁，发现冲突时直接跳过
- **自清理**: 自动清理机制避免锁文件堆积

## 兼容性

- 完全向下兼容现有功能
- 不影响单账号使用场景
- 锁文件为隐藏文件，不干扰正常操作
